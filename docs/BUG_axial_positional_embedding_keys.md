# Bug: axial-positional-embedding Key Naming Change Between v0.2 and v0.3

## Summary

The `axial-positional-embedding` package changed how it registers positional
embedding parameters between versions 0.2.x and 0.3.x. This causes a key
mismatch when loading weights saved with one version into a model built with
the other.

---

## Root Cause

### v0.2.x — custom `ParameterList` with underscore naming

Version 0.2 used a custom mock `ParameterList` class as a workaround for
[PyTorch issue #36035](https://github.com/pytorch/pytorch/issues/36035).
Parameters were registered individually via `setattr()`:

```python
def _keyname(self, prefix, ind):
    return f'{prefix}_{ind}'

def append(self, x):
    setattr(self.kls, self._keyname(self.prefix, self.ind), x)
```

This produces state dict keys like:
- `transformer.axial_pos_emb.weights_0`
- `transformer.axial_pos_emb.weights_1`

### v0.3.x — `nn.ParameterList` with dot naming

Version 0.3 switched to PyTorch's native `nn.ParameterList`:

```python
self.weights = nn.ParameterList([])
```

`nn.ParameterList` uses dot-separated indexing, producing:
- `transformer.axial_pos_emb.weights.0`
- `transformer.axial_pos_emb.weights.1`

---

## Impact

The original BioM3 weights released on HuggingFace were trained with v0.2.1
(per `requirements_orig.txt`). The current project pins v0.3.12. Loading the
original weights into a v0.3 model without correction raises:

```
RuntimeError: Unexpected key(s) in state_dict: "transformer.axial_pos_emb.weights_0", ...
```

---

## Affected code

- `src/biom3/Stage3/io.py` — `_DEFAULT_SUBS` dict handles the bidirectional
  mapping and `prepare_model_ProteoScribe()` applies it when
  `attempt_correction=True`
- `requirements_orig.txt` — pins `axial-positional-embedding==0.2.1`
- `requirements.txt`, `requirements_polaris.txt`, `requirements_spark_py312.txt`
  — pin `axial-positional-embedding==0.3.12`

---

## Key naming reference

| Package version | Key style | Example keys |
|-----------------|-----------|-------------|
| 0.2.x | underscore | `weights_0`, `weights_1` |
| 0.3.x | dot | `weights.0`, `weights.1` |

---

## Test coverage

Dummy weight files in `tests/_data/models/stage3/weights/` exercise both
naming conventions against the correction logic:

| File | Key style | Simulates |
|------|-----------|-----------|
| `origkeys_mini_v2.bin` | underscore (`weights_0`) | Weights saved with v0.2.x |
| `origkeys_mini_v3.bin` | dot (`weights.0`) | Weights saved with v0.3.x |

These are generated by `tests/_scripts/generate_dummy_weights.py` from the
config `tests/_data/models/stage3/configs/origkeys_mini.json` (a miniaturised
version of `orig_model.json` with depth=16 but dim=4, producing all 223 keys
at ~100 KB).

---

## Current fix

The `attempt_correction` parameter in `prepare_model_ProteoScribe()` applies
`_DEFAULT_SUBS` to remap keys bidirectionally. This allows weights from either
version to load correctly regardless of the installed package version.
